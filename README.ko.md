# Exchange Listing Sniper

[English](README.md) · **한국어**

한국 거래소(**업비트** `@upbit_news`, **빗썸** `@BithumbExchange`)의 공식 텔레그램 채널을
감시해, 새 KRW/원화 현물 상장 공지가 뜨는 즉시 감지하고, 같은 티커를 **Bybit 현물
시장가로 즉시 매수**하는 저지연 봇.

논지: 한국 거래소가 **이미 Bybit에서 거래되는** 코인에 대해 **신규 KRW 상장**을
발표하면, 그 공지가 보통 가격을 펌핑시킨다. 봇은 공지 후 수 밀리초 안에 Bybit에서
매수해 그 움직임을 선점한다.

---

> ⚠️ **면책 고지.** 이건 개인 연구/자동화 프로젝트이며 **투자 조언이 아니다**. **실제
> 돈으로 실제 시장가 주문**을 낸다. 암호화폐 거래는 큰 손실 위험을 동반하고, 유동성 낮은
> 신규 상장은 양방향으로 격하게 움직일 수 있으며, 봇에는 **내장 손절·자동매도가 없다**.
> API 키·자금·설정·결과는 전적으로 본인 책임이다. 먼저 **비활성/페이퍼 모드로** 돌리고,
> 작은 금액으로 시작하며, 잃어도 되는 돈만 써라. 어떤 보증도 없다.

## 왜 만들었나

한국의 지배적 거래소(업비트, 빗썸)의 신규 코인 상장은 암호화폐에서 가장 신뢰도 높은
단기 촉매 중 하나다. 한국 현물 시장은 구조적 프리미엄에 거래되고 집중된 리테일 수급으로
움직이기 때문에, 한국 거래소가 *이미* Bybit 같은 글로벌 거래소에서 거래되는 코인에 **신규
KRW 마켓**을 발표하면 그 공지는 즉각적인 매수 쇄도를 일으키고 — Bybit 가격은 같은 분
안에 튄다(아래 백테스트로 정량화).

촉매는 공개돼 있고 움직임은 빠르므로, 유일한 엣지는 **남보다 빠른 것**이다: 공지를
파싱하고 군중보다 먼저 주문을 넣는 것. 이건 재량적 문제가 아니라 *엔지니어링* 문제로,
(1) 텔레그램 메시지를 최소 지연으로 수신하고, (2) "이게 매수 대상 KRW 상장인가?"를
마이크로초 단위로 결정론적으로 판정하고, (3) 캔들이 움직이기 전에 정확하고 멱등한 Bybit
주문을 발사하는 것으로 환원된다. 이 레포가 바로 그 기계다: 각 공지 주위의 몇 초짜리
엣지를 잡기 위한 마이크로초급 감지-매수 파이프라인이며, 실제 과거 데이터로 end-to-end
검증됐다.

## 동작 방식

```
텔레그램 공지
        │  (MTProto, ~ms)
        ▼
  Race 수신  ── TDLib (C++) + Pyrogram + Telethon, 최초 도착 우승, 자동 재연결
        │
        ▼
  최초도착 게이트  ── (채널, message_id) 백엔드 간 중복제거
        │
        ▼
  3중 중복제거  ── replay-floor(재시작) · message-id 윈도우 · 티커별 가드
        │
        ▼
  분류  ── 업비트/빗썸 규칙, Python/C++/Rust/ultra/relay 전부 동일 판정
        │
        ▼
  매수  ── Bybit 현물 시장가 ({TICKER}USDT, quoteCoin), 멱등 orderLinkId
        │
        ▼
  백그라운드 마무리 → 영수증 저장 → 텔레그램 알림
```

핫패스(수신 → 주문준비)는 수 **마이크로초**, 분류는 서브-마이크로초. end-to-end
레이턴시는 거의 전부 **Bybit까지의 네트워크 전파**(~수십 ms; 매칭엔진 근처에
코로케이트하면 최소화)에 의해 좌우된다.

### 무엇이 "매수 대상 상장"인가

- **업비트** — `[거래] …(TICKER) KRW 마켓 디지털 자산 추가` 또는 `…신규 거래지원 안내
  (KRW 마켓)`. 제외: BTC/USDT 단독 추가, 입출금, 유통량, 유의, 이벤트, 상장폐지, 일정변경
  후속.
- **빗썸** — `[마켓 추가] …(TICKER) 원화 마켓 추가` 계열(수수료 이벤트·심볼 리네임 재공지
  포함). 제외: 시간변경, 연기, 입금, 재상장, 유의, 중단, 상장폐지.
- `{TICKER}USDT`가 실제로 Bybit 현물에 존재할 때만 매수; 아니면 스킵(주문 없음, 손실 없음).

## 전략 검증 — 실제 데이터 백테스트

> **100% 실제 데이터, 추정 0.** 공지 시각은 라이브 텔레그램(MTProto) 세션의 *실제* 메시지
> 타임스탬프, 가격은 Bybit 공개 V5 1분봉(OHLC). 어떤 가격·시각도 추론하거나 모델로
> 생성하지 않았다.

**질문.** 빗썸이 이미 Bybit에 있는 코인의 신규 KRW 상장을 공지하면, Bybit가 실제로
펌핑하나 — 얼마나?

**방법.** 90일 구간, 모든 빗썸 `[마켓 추가] …원화 마켓 추가` 공지를 정확한 텔레그램
타임스탬프 기준으로 잡는다. `{TICKER}USDT`가 Bybit 현물에서 거래되면 공지 전후
−5~+10분의 1분봉 OHLC를 가져온다. 진입가는 **공지 분의 시가**("공지 순간에 산다"의
프록시); 윈도우 내 고가(peak), 저가(최악 낙폭), +10분 종가를 측정한다.

**표본.** 구간 내 빗썸 KRW 상장 21건. 13건은 Bybit에 *없어서* 제외 — 봇도 스킵(주문 없음,
손실 없음). 나머지 **8건이 검증 대상.**

| 티커 | 공지(UTC) | 진입가 | +0봉 종가 | Peak (≤+10분) | +10분 종가 | 최악 낙폭 |
| --- | --- | --- | --- | --- | --- | --- |
| VVV | 2026-04-01 03:21 | 6.463 | +4.89% | **+7.18%** | +5.85% | −0.28% |
| ZAMA | 2026-04-14 05:39 | 0.02752 | +8.79% | **+19.91%** | +12.17% | −0.04% |
| BASED | 2026-04-21 05:23 | 0.13111 | +4.28% | **+25.41%** | +15.54% | −0.72% |
| BLEND | 2026-04-29 03:24 | 0.22656 | +5.48% | **+9.85%** | +3.75% | −1.79% |
| OPG | 2026-05-22 01:57 | 0.2491 | +18.75% | **+22.84%** | +15.94% | 0.00% |
| BILL | 2026-05-28 05:16 | 0.08565 | +4.47% | **+7.20%** | +4.73% | −1.05% |
| HNT | 2026-06-01 01:45 | 0.7207 | +12.06% | **+13.21%** | +8.48% | 0.00% |
| SPX | 2026-06-16 02:22 | 0.3628 | +3.94% | **+4.85%** | +2.40% | 0.00% |

**집계 (n = 8).**

- **승률: 8/8 (100%)** — peak·+10분 종가 둘 다 양수.
- **Peak (≤+10분): 평균 +13.81%, 중앙 +11.53%, 최대 +25.41%, 최소 +4.85%.**
- **+10분 보유: 평균 +8.61%, 중앙 +7.16%.**
- 펌핑이 *공지 분 안에서* 터진다 — +0봉 종가가 이미 +4~19% — 한국 수급이 Bybit를 즉시
  끌어올린다는 직접 증거이자 **왜 레이턴시가 엣지인지**의 이유: 빠를수록 펌핑 전 진입가에
  가깝게 잡는다.

**정직한 한계 — 과대 해석 금지.**

- **작은 표본.** n = 8. 8/8은 강한 신호지 보장이 아니다; "sell-the-news", 안 오르는 코인,
  페이크아웃은 언젠가 나온다.
- **진입가는 펌핑 전 가격.** 표의 %는 공지 순간 대비 *상한*이다. 실제 체결은 레이턴시만큼
  펌핑 중간 어딘가라, 실현 수익 < 표 — 그래서 코로케이트한 저지연 호스트가 중요하다.
- **단조 상승 아님.** 8건 중 5건이 인트라바로 진입가 아래를 한 번은 터치(최악 −1.79%).
  엣지는 *비대칭*(작은 하방, 큰 상방)이지 "위로만"이 아니다.
- **수수료·슬리피지 미반영.** 얇은 신규상장 호가에 시장가는 실제 슬리피지 + 왕복 테이커
  수수료를 문다.
- **자동매도 없음.** +8~14%는 고점 근처에서 팔아야 실현; 봇은 사지만 팔지 않고, +10분이면
  일부 반납. 청산은 본인 몫.
- **과거 ≠ 미래.**

데이터셋(`data/bithumb_listing_backtest_90d.json`, 로컬/gitignore)과 위 방법은 Bybit 공개
kline + 봇 자체 텔레그램 세션으로 재현 가능 — 독점 피드 불필요.

### 업비트 교차검증 — 엣지는 *조건부*다

동일 방법을 **업비트**(`@upbit_news`)에 **180일** 적용하면 전혀 다른 그림이 나오고,
엣지의 정체가 분명해진다.

| 마켓 | 표본(Bybit 거래가능) | Peak(≤+10분) 평균 | +10분 보유 평균 | +10분 승률 |
| --- | --- | --- | --- | --- |
| **빗썸** (90일) | 8 | **+13.81%** | **+8.61%** | 8/8 |
| **업비트** (180일) | 28 | +1.19% | +0.13% | 13/28 |

업비트 KRW 상장은 Bybit를 거의 안 움직인다: 평균 peak +1.19%, +10분 보유는 통계적으로
평평(+0.13%, 13/28 양수), 분내 급등 없음, 28건 중 27건이 인트라바로 진입가 밑 터치 — 즉
**수수료·슬리피지 반영하면 이 표본에서 업비트 상장 매수는 순손실.**

왜 갈릴까? 네 코인이 *양쪽* 다 상장됐고, 데이터는 명확하다:

| 코인 | 빗썸(먼저) | 빗썸 peak | 업비트(나중) | 업비트 peak |
| --- | --- | --- | --- | --- |
| ZAMA | 04-14 05:39 | **+19.91%** | 04-14 16:48 | +0.75% |
| BLEND | 04-29 03:24 | **+9.85%** | 04-29 11:23 | +2.09% |
| VVV | 04-01 03:21 | **+7.18%** | 05-12 14:00 | +0.61% |
| SPX | 06-16 02:22 | **+4.85%** | 06-16 11:12 | +0.16% |

모든 경우 빗썸이 **먼저** 상장해 펌핑을 가져갔고, 같은 코인의 나중 업비트 상장은 이미
가격에 반영돼 평평했다. 나머지 24건의 업비트 단독 상장은 이미 자리잡은 대형코인(ICP,
WIF, ETHFI, IO, USDE, TAO, …) 위주로, 한국 상장이 더는 못 움직인다.

**정밀화된 논지.** 촉매는 "한국 상장"이 아니라 **아직 글로벌하게 가격에 반영되지 않은
코인의 *첫* 한국 상장**(보통 작은/신규 코인, 이 구간에선 대개 빗썸)이다. 둘째 한국
상장이나 이미 유동성 충분한 대형코인 상장은 엣지가 거의 없다. *모든* 업비트·빗썸 상장을
사는 봇은 break-even~손실 거래로 진짜 엣지를 희석시킨다; 자연스러운 개선은 타 한국
거래소에 이미 상장된 코인과 자리잡은 대형코인을 스킵하는 것. **정직한 결론: 전략은
실재하나 선택적이다 — "다 줍줍"이 엣지가 아니라 "맞는 상장을, 빠르게, 그리고 청산"이
엣지다.**

## 기능

- **5개 구현으로 입증된 정확성.** Python, C++ 분류기, Rust 분류기, C++ ultra 엔진, TDLib
  relay가 모든 제목에 동일 판정(골든 픽스처 + `bin/verify_listing_classifiers.py`로 강제).
  실제 공지 ~1,000건에 대해 누락 0·오매수 0으로 검증.
- **중복 매수 없음.** 3중 중복제거(replay floor, 제한된 message-id 윈도우, 무제한
  `(채널, 티커)` 가드) + Bybit가 자체적으로 중복을 거부하는 결정론적 `orderLinkId`. 모호한
  전송 타임아웃은 다른 transport로 재발사하지 않는다.
- **머니 세이프티.** 하드 상한(`BYBIT_SPOT_BUY_MAX_USDT_AMOUNT`)이 그 이상이거나 비유한한
  금액을 거부 → 손가락 실수/0 하나 더가 과대 주문을 못 낸다. 멀티코인 상장 시 고정 금액을
  여러 티커에 분할하는 옵션.
- **24/7 안정 수신.** 각 실시간 백엔드가 끊기면 자동 재연결(지수 백오프 + 지터); 배포
  인스턴스는 `systemd Restart=always`로 구동.
- **교체 가능한 주문 transport.** C++ 패스트패스(libcurl), C++ WebSocket, Python
  WebSocket, REST — 선택 및 순차 폴백.

## 기술 스택

Python 3.9 오케스트레이션 · C++20(ultra 엔진, TDLib JSON relay, fast/WS 주문 경로) ·
Rust(분류기) · TDLib + Pyrogram + Telethon(MTProto 수신) · Bybit V5 API.

### 설정 (`.env`)

실시간 **소스**는 봇 토큰이 아니라 텔레그램 **유저** API(`API_ID`/`API_HASH`,
my.telegram.org)가 필요하다. 봇 토큰은 **알림** 전송에만 쓴다.

| 키 | 용도 |
| --- | --- |
| `LISTING_SOURCE_TELEGRAM_API_ID` / `_API_HASH` / `_PHONE` | MTProto 소스 로그인 |
| `BYBIT_API_KEY` / `BYBIT_API_SECRET` | 거래 권한 키(현물 거래; **출금 불가**) |
| `BYBIT_SPOT_BUY_ENABLED` | 라이브 매수 마스터 스위치 |
| `BYBIT_SPOT_BUY_USDT_AMOUNT` | 매수당 견적 금액(USDT) |
| `BYBIT_SPOT_BUY_MAX_USDT_AMOUNT` | 하드 머니세이프티 상한 |
| `LISTING_TELEGRAM_BOT_TOKEN` / `_CHAT_ID` | 알림 전송(선택) |

전체 설정(transport, 네이티브 엔진, readiness 게이트 등)은 `.env.example` 참고.

## 사용법

```bash
# 감지만 — 주문 없음 (안전; 파이프라인 라이브 검증)
.venv/bin/python main.py --realtime --no-trade

# 표준 라이브 (race 백엔드, Python 주문 경로)
.venv/bin/python main.py --realtime

# 최속 모드 (TDLib C++ relay가 직접 주문; readiness 게이트 먼저 실행)
bash bin/run_race_native_buy_realtime.sh

# 단발 단일거래소 폴링 / 테스트 알림
.venv/bin/python main.py --exchange bithumb --no-trade
.venv/bin/python main.py --test-telegram
```

주요 플래그: `--realtime`, `--realtime-backend race|tdlib|telethon|pyrogram`,
`--no-trade`, `--no-telegram`, `--ultra-buy`, `--source-only`, `--strict-realtime`,
`--reset`, `--verbose`. `Ctrl+C`로 중지(graceful flush).

## 테스트

```bash
.venv/bin/python bin/verify_listing_classifiers.py # Python = C++ = Rust = relay 동등성
```

모든 외부 I/O(텔레그램/Bybit/시계/파일시스템)는 모킹돼 스위트가 완전 오프라인·결정론적으로
돈다.

## 24/7 운용

항상 켜진 호스트(Bybit 매칭엔진과 같은 클라우드 리전의 VPS가 주문 레이턴시 최소화)에서
제공된 `systemd` 유닛(`deploy/linux/`)으로 구동하며, `Restart=always`로 프로세스가 자동
복구된다. 실시간 백엔드는 추가로 백엔드별 자동 재연결로 self-heal하므로 일시적 네트워크
끊김이 감시를 멈추지 않는다.
